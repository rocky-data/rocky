# dagster-rocky — Agent Instructions

Dagster integration for the Rocky SQL transformation engine. Wraps the `rocky` CLI binary as a Dagster resource and maps Rocky's discovery/execution output into Dagster assets, checks, and materializations.

## Architecture

Four core layers, each independently usable:

1. **Types** (`types.py` + `types_generated/`) — Pydantic models for every Rocky CLI command's JSON output. Generated models live in `types_generated/` (autogenerated from `../../../schemas/*.schema.json` via `datamodel-code-generator`); `types.py` re-exports them with backward-compat aliases. `parse_rocky_output()` is table-driven and exhaustive.
2. **Resource** (`resource.py`) — `RockyResource` wrapping CLI subprocess calls. Three execution modes: `run()` (buffered), `run_streaming()` (Popen + daemon stderr reader), `run_pipes()` (full Dagster Pipes integration). Configurable `timeout_seconds`, partial-success handling, optional HTTP fallback for compile/lineage/metrics via `server_url`.
3. **Translator** (`translator.py`) — Maps Rocky metadata to Dagster asset keys, groups, tags. Subclass to customize.
4. **Component** (`component.py`) — State-backed Dagster component with cached discovery. Built from small helpers (`_load_state`, `_build_group_contexts`, `_select_filters`, `_run_filters`, `_emit_results`, `_emit_placeholder_checks`).

Feature modules:
- `assets.py` — `load_rocky_assets()` functional API
- `checks.py` — `emit_materializations()`, `emit_check_results()`, `check_metadata()`, `cost_metadata_from_optimize()`
- `automation.py` — Dagster automation rules from Rocky config
- `branch_deploy.py` — Branch deployment support
- `column_lineage.py` — Column-level lineage mapping
- `contracts.py` — Data contract enforcement
- `derived_models.py` — Silver-layer derived model assets
- `freshness.py` — FreshnessPolicy projection
- `health.py` — Health check integration
- `observability.py` — Observability event helpers
- `partitions.py` — Time-partitioned asset support
- `scaffold.py` — Project scaffolding utilities
- `schedules.py` — Schedule definitions
- `sensor.py` — Sensor definitions (freshness, drift)

## Coding Standards

- Python 3.11+, `from __future__ import annotations` in all modules
- Pydantic `BaseModel` for all data structures
- Type hints on all public functions
- Line length: 100 characters
- Ruff rules: E, F, I, N, UP, B, SIM

## Testing

All tests use scenario data from `scenarios.py` (Python dicts) — no Rocky binary or credentials needed. Live-binary fixtures in `fixtures_generated/` are exercised by `test_generated_fixtures.py`.

## Key Design Decisions

- **State-backed component**: `rocky discover` cached to avoid repeated CLI calls
- **Partial success**: Rocky can exit non-zero but return valid JSON
- **Subset-aware**: Multi-asset `can_subset=True` for selective execution. The component drops materializations and checks for asset keys outside the requested subset, and drops check kinds the component never declared, to satisfy Dagster's selection invariants.
- **HTTP fallback**: Supports `server_url` for live Rocky server mode
- **Phase 2 codegen**: Types are autogenerated from JSON Schema via `just codegen-dagster`. `types.py` re-exports with backward-compat aliases.
