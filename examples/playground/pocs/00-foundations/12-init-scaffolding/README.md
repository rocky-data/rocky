# 12-init-scaffolding — Project scaffolding with `rocky init`

> **Category:** 00-foundations
> **Credentials:** none (DuckDB)
> **Runtime:** < 10s
> **Rocky features:** `rocky init`, project templates

## What it shows

`rocky init` scaffolds a new Rocky project from a template. The default
`duckdb` template lays down a `rocky.toml`, a `models/` directory with a sample
transformation model and shared defaults, and a `seeds/seed.sql` — a project
that validates and compiles immediately, with no credentials. The `--template`
flag selects a different starting point (`duckdb`, `databricks-fivetran`,
`snowflake`).

This POC scaffolds a DuckDB project into `./scaffolded/`, prints the tree, runs
the exact quickstart the scaffolder recommends (`validate` + `compile`) to prove
it works out of the box, then scaffolds a Snowflake project to show the template
flag changes the layout.

## Why it's distinctive

- The scaffolded project is **runnable on the first command** — `rocky validate`
  and `rocky compile` pass against a fresh `rocky init` with zero edits.
- One flag swaps the entire starting config between warehouses; you are not
  hand-assembling adapter blocks to get going.

## Layout

```
.
├── README.md         this file
├── rocky.toml        placeholder (the demo scaffolds its own project at runtime)
└── run.sh            init → show tree → validate + compile → init a second template
```

The `./scaffolded/` and `./scaffolded-snowflake/` directories are generated by
`run.sh` and gitignored — they are regenerated on every run.

## Prerequisites

- `rocky` on PATH
- `duckdb` CLI for seeding (`brew install duckdb`)
- `jq` for reading the JSON output

## Run

```bash
./run.sh
```

## Expected output

```text
==> 1. rocky init — scaffold a fresh DuckDB project into ./scaffolded/
Initialized Rocky project in scaffolded

  rocky.toml           — pipeline config (DuckDB)
  models/_defaults.toml — shared model defaults
  models/stg_orders.*  — sample transformation model
  seeds/seed.sql       — sample data

==> 2. Scaffolded tree:
    ./models/_defaults.toml
    ./models/stg_orders.sql
    ./models/stg_orders.toml
    ./rocky.toml
    ./seeds/seed.sql

==> 3. Run the quickstart the scaffold prints — it works out of the box.
      valid : true
      models compiled : 1
```

## What happened

1. `rocky init scaffolded --template duckdb` wrote a complete starter project.
2. The tree shows the config, sample model, model defaults, and seed file.
3. Running the scaffold's own quickstart — seed, `rocky validate`,
   `rocky compile` — passes immediately on the freshly generated project.
4. `rocky init --template snowflake` scaffolds a Snowflake-shaped project,
   showing the template flag changes what you start from.
