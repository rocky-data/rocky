# Development Guide

## Prerequisites

- **Node.js 20+** and **npm**
- **VS Code 1.85.0+**
- **[Rocky CLI](https://github.com/rocky-data/rocky)** вЂ” installed and on `$PATH` (the extension spawns it as an LSP server)
- **TypeScript 5.4+** (installed as devDependency)

## Getting Started

```bash
git clone https://github.com/rocky-data/rocky.git
cd rocky/editors/vscode
npm install
npm run compile
```

Then press **F5** in VS Code to launch the Extension Development Host with the extension loaded.

## Architecture

### LSP Client (`lspClient.ts`)

The extension is primarily an LSP client. On activation, it spawns the Rocky CLI as a language server:

```
rocky lsp [extraArgs...]
```

Communication happens over **stdio** (JSON-RPC). The binary path and extra args are configurable via settings (`rocky.server.path`, `rocky.server.extraArgs`).

**Document selector**: `.rocky` files and `.sql` files under `**/models/**` directories.

**File watchers**: `**/*.rocky`, `**/*.toml`, `**/models/**/*.sql` вЂ” the server is notified of changes to these files.

The LSP server provides: diagnostics, hover, go-to-definition, find references, rename, completion, signature help, document symbols, code actions, inlay hints, and semantic tokens.

### Status Bar

A left-aligned status bar item shows server state (Starting/Ready/Failed) and a live error count from diagnostics. Clicking it restarts the server.

### Commands

The extension registers 50 commands organized by concern under `src/commands/` (one file per group, registered in `src/commands/index.ts`). The full list with descriptions lives in `CLAUDE.md`; the highlights below cover the most common workflows:

| Command | What it does |
|---------|-------------|
| `rocky.restartServer` | Stops and restarts the LSP server |
| `rocky.showLineage` | Renders model DAG as an interactive SVG (webview) |
| `rocky.compile` | Type-check models, validate contracts |
| `rocky.validate` | Validate config and models without API calls |
| `rocky.ci` | Combined compile + test for CI |
| `rocky.test` | Run local tests via DuckDB |
| `rocky.run` | Execute the full pipeline |
| `rocky.plan` | Preview SQL (dry-run) |
| `rocky.discover` | List connectors and tables |
| `rocky.compare` | Shadow vs production comparison |
| `rocky.history` | Run history with trend analysis |
| `rocky.metrics` | Quality metrics with alerts |
| `rocky.compact` | Generate OPTIMIZE/VACUUM SQL |
| `rocky.archive` | Partition archival |
| `rocky.profileStorage` | Column encoding recommendations |
| `rocky.aiGenerate` | Creates a model from natural language intent |
| `rocky.aiSync` | Detects schema changes and proposes model updates |
| `rocky.aiExplain` | Generates intent documentation from a model |
| `rocky.aiTest` | Generates test cases from model intent |
| `rocky.importDbt` | Convert dbt project to Rocky |
| `rocky.validateMigration` | Compares a dbt project against Rocky models |
| `rocky.hooksList` | List configured hooks |
| `rocky.hooksTest` | Fires a test hook event (quick-pick selector) |
| `rocky.doctor` | Runs health checks, displays results |
| `rocky.optimize` | Cost and strategy analysis, opens JSON results |

Commands use `cp.execFile()` (not `exec()`) for safe subprocess execution and `vscode.window.withProgress()` for long-running operations.

### Lineage Webview

The `rocky.showLineage` command extracts the model name from the current file, calls `rocky lineage <model> --format dot`, and renders the DOT graph as SVG using viz.js. The webview uses VS Code CSS variables for theme-aware styling.

## Development Workflow

There are three ways to run the extension during development:

### 1. Extension Development Host (recommended for code changes)

Press **F5** in VS Code (or Run > Start Debugging). This opens a new VS Code window with the extension loaded from your working directory. Changes to TypeScript require recompiling (`npm run compile` or use `npm run watch`) and reloading the dev host (Ctrl+Shift+P > "Developer: Reload Window").

The debug configuration is in `.vscode/launch.json`.

### 2. Install from .vsix (for testing the packaged extension)

```bash
npm run package
code --install-extension rocky-0.2.0.vsix
```

Best for verifying that packaging works correctly and nothing is missing from the bundle.

### 3. Symlink (fastest for grammar/snippet/schema changes)

```bash
ln -s "$(pwd)" ~/.vscode/extensions/rocky-dev.rocky-0.2.0
```

Changes to JSON files (grammar, snippets, schemas, language config) take effect on reload without recompilation. Remove the symlink when done.

## Testing

Two test suites:

1. **Vitest unit tests** (`src/__tests__/`) вЂ” fast, no VS Code instance needed. Run with `npm run test:unit`.
2. **Electron integration tests** (`src/test/suite/`) вЂ” Mocha with TDD UI, runs inside a real VS Code instance via `@vscode/test-electron`. Downloads ~344 MB on first run. Run with `npm test`.

```bash
# Fast unit tests (default for `just test-vscode`)
npm run test:unit

# Full electron integration tests
npm run compile && npm test
```

**Current test suite** (`src/test/suite/extension.test.ts`):
- Extension is present and activatable
- All expected commands are registered
- Rocky language is registered

Tests run with a 10-second global timeout and 1-second per-test timeout.

**Adding tests**: Create new `.test.ts` files in `src/test/suite/`. The test loader discovers all `**/*.test.js` files automatically.

## Extension Manifest

Key sections of `package.json` that affect behavior:

### Activation Events

The extension activates when:
- A `.rocky` file is opened (`onLanguage:rocky`)
- The workspace contains `*.rocky` files or `rocky.toml`

### Settings (contributes.configuration)

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `rocky.server.path` | string | `"rocky"` | Path to the Rocky binary |
| `rocky.server.extraArgs` | array | `[]` | Extra arguments passed to `rocky lsp` |
| `rocky.inlayHints.enabled` | boolean | `true` | Show inferred types inline |

### Language Registration

- Language ID: `rocky`
- File extensions: `.rocky`
- Grammar: TextMate (`syntaxes/rocky.tmLanguage.json`)
- Snippets: `snippets/rocky.json`
- Config schema: validates `*.rocky.toml` files against `schemas/rocky-config.schema.json`

## Modifying the Grammar

The TextMate grammar is at `syntaxes/rocky.tmLanguage.json`. It defines syntax highlighting for:

- Comments (`--`)
- Pipeline steps (`from`, `where`, `group`, `derive`, `select`, `join`, `sort`, `take`, `distinct`, `replicate`)
- Keywords (`as`, `on`, `keep`, `asc`, `desc`, join types)
- Literals (strings, numbers, dates `@YYYY-MM-DD`, booleans, null)
- Operators (logical, comparison, arithmetic, arrow `=>`)
- Functions (aggregate, string, numeric, datetime, window)
- Match expressions (case/when)

When the LSP server is running, it also provides **semantic tokens** that overlay the TextMate grammar with richer, context-aware highlighting. The semantic token colors are defined in `themes/rocky-semantic.json`.

## Adding Snippets

Snippets are in `snippets/rocky.json`. Each snippet has a prefix (trigger text), body (template with tab stops), and description. Use VS Code snippet syntax вЂ” `$1`, `$2` for tab stops, `${1|choice1,choice2|}` for choices.

## Packaging

```bash
npm run package
```

This creates `rocky-<version>.vsix`. The `.vscodeignore` file excludes source files, source maps, node_modules, and config files from the package вЂ” only compiled JS, JSON configs, and assets are included.

- **Module**: CommonJS
- **Strict mode**: enabled
- **Source maps**: enabled (for debugging)
